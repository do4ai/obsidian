# 2. 성능, 보안, 권한 점검
이전 페이지에서는 구매자, 관리자, AI 상담 챗봇을 하나의 E2E 흐름으로 검증했습니다. 이번 페이지에서는 그 흐름이 실제 서비스로 운영될 수 있는 품질을 갖췄는지 확인합니다. 여기서 품질은 단순히 "빠르다"가 아닙니다. 응답이 예측 가능하고, 민감 정보가 새지 않으며, 권한 경계가 백엔드에서 강제되는 상태를 말합니다.

실전 이커머스에서는 기능이 동작해도 성능과 보안과 권한이 불안하면 출시할 수 없습니다. 상품 목록이 느리면 구매자는 이탈하고, 주문 권한이 흔들리면 개인정보 사고가 됩니다. admin 권한이 프론트에서만 막혀 있으면 누구든 API를 직접 호출해 운영 데이터를 볼 수 있습니다.

즉 이번 페이지는 기능 완료 이후 릴리스 전 품질 문턱을 세우는 단계입니다.

# 왜 성능, 보안, 권한을 같이 보는가
세 항목은 따로 떨어져 보이지만 실제 서비스에서는 같이 움직입니다.

- 성능을 위해 데이터를 많이 내려주면 민감 정보가 섞일 수 있다
- 권한 검사를 생략하면 빠르지만 위험하다
- 보안 로그를 남기지 않으면 장애와 공격을 구분하기 어렵다
- 챗봇 응답 품질을 높이려다 프롬프트에 과도한 개인정보를 넣을 수 있다

그래서 이번 장에서는 성능 최적화, 보안 점검, 권한 검사를 하나의 릴리스 점검 흐름으로 다룹니다.

# 이 페이지에서 정할 것
이 페이지를 읽고 나면 아래 항목이 정리되어 있어야 합니다.

- 어떤 화면과 API를 성능 점검 대상으로 볼 것인가
- 응답 시간과 데이터 크기는 어느 정도를 목표로 할 것인가
- 민감 정보는 어떤 응답과 로그에서 제거해야 하는가
- buyer, admin, 비회원 권한은 어디서 강제해야 하는가
- 챗봇은 어떤 정보를 프롬프트에 넣으면 안 되는가
- 릴리스 전 반드시 막아야 할 위험은 무엇인가

# 1. 성능 점검은 어디서 시작하는가
처음부터 모든 API를 깊게 튜닝할 필요는 없습니다. 사용자가 자주 지나가는 경로부터 봅니다.

우선 점검 대상은 아래입니다.

- 상품 목록 API
- 상품 상세 API
- 로그인과 현재 사용자 확인 API
- 장바구니 조회와 수량 변경 API
- 주문 생성 API
- 결제 확인 API
- admin 주문 목록 API
- 챗봇 상담 API

이 API들은 구매 흐름과 운영 흐름의 핵심입니다. 여기서 병목이 생기면 서비스 전체가 느리게 느껴집니다.

# 2. 화면 성능은 무엇을 봐야 하는가
프론트 화면에서는 단순히 "로딩이 된다"보다 아래 항목을 봅니다.

- 첫 화면에서 필요한 데이터만 요청하는가
- 같은 데이터를 불필요하게 반복 요청하지 않는가
- loading, empty, error 상태가 즉시 보이는가
- admin 목록이 커져도 화면이 멈추지 않는가
- 챗봇 응답 대기 중 사용자가 상태를 이해할 수 있는가

특히 admin 화면은 데이터가 늘어날수록 느려지기 쉽습니다. 목록 API는 처음부터 pagination이나 limit 기준을 염두에 두는 편이 좋습니다.

# 3. API 응답 목표는 어떻게 잡는가
MVP에서도 대략적인 목표는 있어야 합니다. 아래 기준으로 시작할 수 있습니다.

- 상품 목록: 로컬 기준 300ms 안쪽
- 상품 상세: 로컬 기준 200ms 안쪽
- 장바구니 조회/수정: 로컬 기준 300ms 안쪽
- 주문 생성: 로컬 기준 500ms 안쪽
- admin 목록: 로컬 기준 500ms 안쪽
- 챗봇 응답: LLM 제외 내부 처리 500ms 안쪽

실제 운영 환경에서는 네트워크와 외부 API 때문에 달라질 수 있습니다. 중요한 것은 목표 숫자보다, 느려졌을 때 비교할 기준선을 남기는 것입니다.

# 4. DB와 쿼리 병목은 어떻게 찾는가
이커머스에서 흔한 병목은 목록 조회와 관계 데이터 로딩입니다.

우선 아래를 확인합니다.

- 상품 목록이 모든 컬럼을 가져오지 않는가
- 주문 목록에서 주문 항목을 과도하게 함께 가져오지 않는가
- 사용자별 장바구니 조회에 인덱스가 필요한가
- 주문 ID 조회에 소유자 조건이 함께 붙는가
- admin 목록은 limit과 offset 또는 cursor가 있는가

초기에는 복잡한 최적화보다 쿼리 수와 응답 크기를 줄이는 것이 효과적입니다.

# 5. 캐싱은 언제 생각하는가
캐싱은 성능을 올릴 수 있지만, 이커머스에서는 데이터 신선도가 중요합니다.

캐싱 후보는 아래 정도입니다.

- 공개 상품 목록
- 배송/교환/환불 정책 문서
- FAQ 검색 결과
- 비개인화 추천 질문

캐싱을 조심해야 하는 대상은 아래입니다.

- 사용자별 장바구니
- 주문 상태
- 결제 상태
- admin 운영 데이터
- 챗봇의 사용자별 상담 응답

사용자별 데이터는 잘못 캐시하면 다른 사용자 정보가 노출될 수 있습니다.

# 6. 보안 점검은 민감 정보부터 시작한다
보안 점검의 첫 단계는 민감 정보가 어디에 흐르는지 확인하는 것입니다.

이번 프로젝트에서 민감 정보 후보는 아래입니다.

- 비밀번호
- access token 또는 session token
- 사용자 이메일
- 배송지와 연락처
- 결제 수단 원문
- 주문 상세 정보
- 상담 대화 원문

이 정보들은 API 응답, 로그, 브라우저 저장소, LLM 프롬프트, admin 화면에서 각각 어떻게 다뤄지는지 확인해야 합니다.

# 7. 로그에 남기면 안 되는 것은 무엇인가
디버깅을 위해 모든 요청을 찍는 습관은 위험합니다. 특히 아래 정보는 로그에 그대로 남기지 않는 편이 좋습니다.

- 비밀번호
- 인증 토큰
- 결제 수단 상세
- 전체 배송 주소
- 상담 원문 중 민감 정보
- LLM 프롬프트 전체

대신 로그에는 요청 ID, 사용자 ID 일부, 상태 코드, 처리 시간, 실패 유형처럼 추적에 필요한 최소 정보만 남깁니다.

# 8. 권한 검사는 어디에서 강제해야 하는가
권한 검사는 반드시 백엔드에서 강제해야 합니다. 프론트의 보호 라우트는 사용자 경험을 위한 장치일 뿐입니다.

이번 프로젝트의 권한 원칙은 아래입니다.

- 비회원은 공개 상품만 조회할 수 있다
- buyer는 자신의 장바구니와 주문만 볼 수 있다
- buyer는 admin API에 접근할 수 없다
- admin은 운영 데이터에 접근할 수 있다
- 챗봇은 사용자 권한 안의 데이터만 참고한다

이 원칙은 라우터, 애플리케이션 서비스, 테스트에서 모두 확인되어야 합니다.

# 9. buyer 권한은 무엇을 막아야 하는가
buyer 권한 점검에서는 아래를 확인합니다.

- 다른 사용자의 주문 ID로 조회할 수 없는가
- 다른 사용자의 장바구니를 수정할 수 없는가
- admin dashboard API를 호출할 수 없는가
- 판매 중지 상품을 주문할 수 없는가
- 품절 상품 수량을 강제로 주문할 수 없는가

이 검증은 프론트 화면이 아니라 API 직접 호출 기준으로도 확인해야 합니다.

# 10. admin 권한은 무엇을 조심해야 하는가
admin은 강한 권한을 가지므로 더 조심해야 합니다.

확인할 항목은 아래입니다.

- admin 로그인과 buyer 로그인이 명확히 구분되는가
- admin API가 role을 확인하는가
- 상품 가격과 재고 수정이 기록되는가
- 주문 상세에서 필요한 정보만 보여주는가
- admin 화면 오류가 내부 시스템 정보를 노출하지 않는가

admin이라고 해서 모든 민감 정보를 그대로 보여줘도 되는 것은 아닙니다.

# 11. 챗봇 보안은 무엇이 다른가
챗봇은 사용자가 자유롭게 입력하기 때문에 일반 폼보다 예측하기 어렵습니다.

챗봇 점검 항목은 아래입니다.

- 다른 사용자의 주문을 물어보면 거절하는가
- 프롬프트에 토큰이나 결제 정보를 넣지 않는가
- 정책 문서에 없는 내용을 단정하지 않는가
- 민감 정보가 포함된 사용자 메시지를 로그에 어떻게 저장하는가
- handoff 시 운영자에게 필요한 맥락만 넘기는가

챗봇은 LLM이 답변을 생성하므로, 권한과 데이터 제한은 LLM 호출 전에 백엔드에서 끝나야 합니다.

# 12. 입력 검증은 어디서 필요한가
입력 검증은 프론트와 백엔드 모두에서 필요합니다.

프론트는 사용자에게 빠르게 알려주는 역할을 합니다.

- 수량은 1 이상이어야 한다
- 이메일 형식이 맞아야 한다
- 필수 배송 정보가 비어 있으면 안 된다
- 챗봇 메시지가 너무 길면 제한한다

백엔드는 실제 신뢰 경계입니다.

- 수량과 가격을 서버 기준으로 다시 계산한다
- 주문 생성 시 상품 상태와 재고를 다시 확인한다
- role과 소유자를 검사한다
- 허용되지 않은 필드는 무시하거나 거절한다

프론트 검증은 편의이고, 백엔드 검증은 필수입니다.

# 13. 취약한 흐름은 어떻게 우선순위를 매기는가
모든 위험을 한 번에 해결하려고 하면 진행이 느려집니다. 우선순위는 피해 규모와 발생 가능성으로 봅니다.

가장 먼저 막아야 할 것은 아래입니다.

- 다른 사용자 주문 조회
- admin API 무단 접근
- 주문 금액 조작
- 결제 성공 상태 위조
- LLM 프롬프트에 민감 정보 포함
- 상품 가격 변경으로 과거 주문 금액 변경

이 항목은 릴리스 전 반드시 검증해야 합니다.

# 14. 성능과 보안 결과는 어떻게 기록하는가
점검 결과는 말로만 남기지 말고 재현 가능한 형태로 남깁니다.

각 항목에는 아래를 기록합니다.

- 점검 대상
- 점검 방법
- 관찰 결과
- 통과 기준
- 미해결 위험
- 후속 작업

이 기록이 있어야 다음 릴리스에서 같은 항목을 다시 확인할 수 있습니다.

# 15. 이번 장의 구현 완료 기준은 무엇인가
이번 장에서 완료라고 볼 수 있는 상태는 아래 정도입니다.

- 핵심 API 응답 기준선이 있다
- 목록 API의 pagination 또는 limit 기준이 있다
- 민감 정보 목록과 노출 금지 위치가 정리되어 있다
- buyer/admin/비회원 권한 원칙이 명확하다
- 권한 실패 테스트가 있다
- 챗봇 프롬프트와 로그의 민감 정보 기준이 있다
- 릴리스 전 차단해야 할 위험 목록이 있다

즉 이제 서비스는 동작 여부를 넘어, 출시 전 품질 문턱을 통과할 준비를 하게 됩니다.

# 16. 이번 페이지를 실제 구현 산출물로 바꾸면 무엇이 나와야 하나
이번 페이지를 코드와 실행 결과로 옮기면 최소한 아래 산출물이 있어야 합니다.

- API 응답 시간 측정 결과
- 핵심 API 권한 테스트
- 다른 사용자 주문 접근 차단 테스트
- admin API role 검사 테스트
- 주문 금액 서버 재계산 테스트
- 민감 정보 로그 제외 규칙
- 챗봇 프롬프트 데이터 제한 규칙
- 릴리스 전 보안 점검표

이 산출물이 있어야 다음 페이지에서 로그, 모니터링, 운영 점검표를 실제 운영 기준으로 정리할 수 있습니다.

# 17. 자주 하는 실수
- 프론트 보호 라우트만 믿고 백엔드 권한 검사를 빼먹는다
- 주문 금액을 클라이언트가 보낸 값으로 믿는다
- admin API가 buyer 토큰으로도 호출되는지 확인하지 않는다
- 로그에 토큰, 배송지, 상담 원문을 그대로 남긴다
- 챗봇 프롬프트에 너무 많은 사용자 데이터를 넣는다
- 상품 목록 성능만 보고 admin 목록 성능을 놓친다
- 캐싱으로 사용자별 데이터를 섞이게 만든다

이 일곱 가지 실수는 기능이 모두 구현된 뒤에도 출시를 막는 원인이 됩니다.

# 정리
이번 페이지에서는 통합 E2E 이후 확인해야 할 성능, 보안, 권한 점검 기준을 정리했습니다. 핵심은 빠르게 만드는 것이 아니라, 어떤 데이터가 누구에게 보이고 어떤 API가 어떤 권한에서 실행되는지 백엔드 기준으로 확정하는 것입니다.
