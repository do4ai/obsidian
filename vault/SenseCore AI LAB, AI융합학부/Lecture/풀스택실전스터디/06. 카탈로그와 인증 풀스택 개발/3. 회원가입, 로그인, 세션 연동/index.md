# 3. 회원가입, 로그인, 세션 연동
앞 페이지에서 `storefront`에 상품 목록과 상세를 붙였다면, 이제 사용자를 실제로 식별하는 흐름을 완성해야 합니다. 상품 탐색만 가능한 서비스는 데모로는 충분할 수 있지만, 이커머스 서비스로 이어지려면 결국 사용자가 누구인지 알아야 합니다. 장바구니, 주문, 결제, 주문 내역 조회는 모두 사용자 식별 위에 세워지기 때문입니다.

이번 페이지의 목적은 회원가입 화면과 로그인 화면을 만드는 것에 그치지 않습니다. 백엔드의 `Identity` API를 화면에 연결하고, 로그인 성공 후 세션을 어떻게 저장하고 복원할지, 인증 실패와 만료 상태를 어떻게 다룰지, 로그인 후 어떤 경로로 이동할지를 함께 정하는 것입니다.

즉 이번 장은 "폼 하나 더 만든다"가 아니라, `storefront`를 비회원 탐색 앱에서 인증 가능한 사용자 앱으로 바꾸는 단계입니다.

# 왜 카탈로그 다음에 인증을 붙이는가
이번 교재의 사용자 여정은 아래 흐름으로 보는 것이 자연스럽습니다.

- 상품 목록을 본다
- 상품 상세를 본다
- 더 진행하려고 하면 로그인 또는 회원가입이 필요해진다
- 로그인 후 개인화된 기능으로 넘어간다

이 순서는 실제 사용자 경험에도 맞고, 학습 순서에도 맞습니다. 상품이 보이기 전에는 로그인의 이유가 약하고, 로그인 후 어디로 가야 할지도 불분명합니다. 반대로 카탈로그가 살아 있으면 인증은 "다음 행동을 여는 관문"으로 자연스럽게 이해됩니다.

# 이 페이지에서 정할 것
이 페이지를 읽고 나면 아래 항목이 정리되어 있어야 합니다.

- 회원가입 화면과 로그인 화면을 어떤 역할로 나눌 것인가
- `signup`, `login`, `me` API를 어떻게 연결할 것인가
- 세션 정보는 어디에 저장하고 언제 복원할 것인가
- 인증 실패와 입력 오류는 어떻게 표현할 것인가
- 로그인 전후 라우트 전이는 어떻게 설계할 것인가

# 1. 이번 단계에서 인증 흐름이 반드시 해야 할 일
아직 주문까지 가지 않더라도, `storefront`의 인증 흐름은 최소한 아래를 만족해야 합니다.

- 회원가입이 된다
- 로그인이 된다
- 새로고침 후에도 인증 상태를 복원할 수 있다
- 현재 로그인한 사용자를 확인할 수 있다
- 인증 실패 시 적절한 메시지를 보여준다

즉 이번 장의 목표는 "로그인 버튼이 있다"가 아니라, 실제로 인증 상태가 앱에 반영된다는 것입니다.

# 2. 회원가입 화면의 첫 책임은 무엇인가
회원가입 화면은 단순한 입력 폼이 아닙니다. 새 사용자를 시스템에 등록하는 첫 관문이므로 아래 정도에 집중하는 것이 좋습니다.

- 필수 입력 필드
- 입력 검증
- 제출 성공 시 다음 행동 안내
- 중복 식별자 오류 처리

초기 단계에서는 마케팅 수신 동의나 세부 프로필까지 넣을 필요는 없습니다. 다음 장의 흐름을 지탱할 최소 가입 정보가 우선입니다.

# 3. 로그인 화면의 첫 책임은 무엇인가
로그인 화면은 사용자에게 가장 자주 실패를 경험하게 하는 화면이기도 합니다. 따라서 아래 역할을 먼저 분명히 하는 것이 좋습니다.

- 로그인 식별자 입력
- 비밀번호 입력
- 제출 중 상태 표시
- 실패 메시지 표시
- 성공 후 다음 경로 이동

이 화면은 예쁘게 보이는 것보다, 실패를 명확히 이해시키고 성공 후 자연스럽게 이동시키는 것이 더 중요합니다.

# 4. 회원가입과 로그인은 왜 분리해서 생각해야 하는가
같은 인증 영역처럼 보여도 회원가입과 로그인은 목적이 다릅니다.

- 회원가입은 새 계정 생성
- 로그인은 기존 계정 확인

이 둘을 한 화면에 섞어도 되지만, 교재 기준으로는 책임을 분리해 설명하는 편이 낫습니다. 그래야 API도 명확해지고, 실패 상황도 अलग-अलग 다룰 수 있습니다. 또한 이후 운영자 인증과도 구분하기 쉬워집니다.

# 5. 어떤 API를 어떤 화면에 연결해야 하는가
이번 장에서는 백엔드에서 이미 준비한 아래 API가 프론트 흐름으로 연결됩니다.

- `POST /auth/signup`
- `POST /auth/login`
- `GET /auth/me`

연결 순서는 아래처럼 보는 것이 자연스럽습니다.

- 회원가입 화면 -> `signup`
- 로그인 화면 -> `login`
- 앱 초기화 또는 새로고침 시 -> `me`

이 순서가 잡혀 있어야 세션 저장과 복원 흐름도 자연스럽게 이어집니다.

# 6. 세션 저장은 왜 로그인 성공 직후만 생각하면 안 되는가
로그인 구현에서 자주 놓치는 것이 "성공 후 저장"만 생각하고 "다음 실행 시 복원"을 빼먹는 점입니다. 하지만 실제 사용자 경험에서는 아래 둘이 모두 필요합니다.

- 로그인 직후 세션 반영
- 앱 재시작 또는 새로고침 후 세션 복원

둘 중 하나라도 빠지면 보호 라우트와 사용자 표시가 쉽게 어긋납니다. 따라서 이번 교재에서는 세션 저장과 복원을 한 세트로 봐야 합니다.

# 7. 세션은 어디에서 관리해야 하는가
세션 정보는 페이지 컴포넌트마다 따로 다루면 안 됩니다. 이번 교재에서는 앞서 정한 provider 구조를 유지하면서, 인증 상태는 앱 전역에서 읽을 수 있게 두는 편이 좋습니다.

예를 들어 아래 정도 상태가 필요합니다.

- 현재 사용자 정보
- 인증 여부
- 초기 복원 중 여부
- 로그인/로그아웃 동작

즉 세션은 한 화면의 상태가 아니라 앱 전역 흐름입니다.

# 8. `me` API는 왜 중요한가
`login`만 있으면 인증이 끝난 것처럼 보이지만, 실제 앱은 현재 사용자 확인이 따로 필요합니다. `GET /auth/me`는 아래 역할을 합니다.

- 새로고침 후 사용자 복원
- 보호 라우트 진입 판단
- 헤더나 메뉴에 사용자 상태 표시
- 세션 만료 여부 확인

즉 `me` API는 세션 복원의 기준점입니다. 이 단계에서 빠지면 로그인은 되지만 앱 전체 인증 상태는 쉽게 불안정해집니다.

# 9. 로그인 성공 후 어디로 이동해야 하는가
로그인 성공 후 이동은 단순히 홈으로 보내는 것으로 끝내면 안 됩니다. 사용자가 어디에서 로그인했는지가 중요할 수 있기 때문입니다.

예를 들어 다음 흐름을 생각할 수 있습니다.

- 상품 상세에서 로그인 유도 -> 로그인 성공 후 다시 상세로 복귀
- 직접 로그인 화면 진입 -> 로그인 성공 후 기본 홈 또는 이전 경로 이동

이 기준이 있으면 다음 장에서 장바구니나 구매 전이도 자연스럽게 붙습니다.

# 10. 입력 오류와 인증 오류는 어떻게 구분해야 하는가
인증 화면에서 실패는 크게 두 종류입니다.

- 입력 자체가 잘못된 경우
- 인증 정보가 틀린 경우

예를 들어 아래처럼 다뤄야 합니다.

- 이메일 형식 오류나 비밀번호 길이 부족은 입력 오류
- 계정 없음, 비밀번호 불일치는 인증 오류

이 둘을 같은 메시지로 뭉개면 사용자는 무엇을 고쳐야 하는지 알기 어렵습니다.

# 11. 로딩 상태는 어떻게 보여줘야 하는가
인증 화면은 제출 직후 상태가 특히 중요합니다. 아래 정도는 최소한 보여야 합니다.

- 제출 버튼 disabled 또는 loading 표시
- 중복 제출 방지
- 성공 또는 실패 결과가 오기 전까지 진행 상태 유지

로그인 버튼을 여러 번 눌러 중복 요청이 나가는 경험은 기본적인 UX 결함이므로, 이 단계에서부터 막는 것이 좋습니다.

# 12. 세션 만료 또는 인증 실패는 앱에서 어떻게 반응해야 하는가
세션은 영원하지 않습니다. 따라서 다음 상황도 미리 생각해야 합니다.

- `me` 요청 실패
- 보호 API 호출 시 401 응답
- 저장된 세션이 더 이상 유효하지 않은 경우

이번 교재에서는 최소한 아래 행동을 먼저 정하는 것이 좋습니다.

- 세션 상태 정리
- 로그인 화면으로 이동 또는 재로그인 유도
- 사용자에게 이해 가능한 메시지 표시

이 기준이 있어야 다음 장의 보호 라우트와 admin 접근 제어도 일관되게 붙습니다.

# 13. 회원가입 성공 후 흐름은 어떻게 설계할까
회원가입 성공 후에는 보통 두 가지 선택지가 있습니다.

- 바로 로그인된 상태로 전환
- 로그인 화면으로 보내거나 다시 로그인하게 함

이번 교재에서는 어느 방식을 택하든, 한 가지 기준으로 일관되게 설명하는 것이 중요합니다. 초기 학습 흐름상으로는 "회원가입 성공 -> 로그인 진입 또는 자동 인증 여부를 명확히 정의"하는 편이 좋습니다.

# 14. 이번 장의 구현 완료 기준은 무엇인가
이번 장에서 완료라고 볼 수 있는 상태는 아래 정도입니다.

- 회원가입 화면이 실제 API와 연결된다
- 로그인 화면이 실제 API와 연결된다
- 로그인 성공 후 세션 상태가 앱에 반영된다
- 새로고침 후 `me`로 세션 복원이 된다
- 입력 오류와 인증 오류가 구분되어 보인다

즉 단순한 auth form이 아니라, 실제 사용자 식별 흐름이 앱에 연결된 상태여야 합니다.

# 15. 이번 페이지를 실제 구현 산출물로 바꾸면 무엇이 나와야 하나
이번 페이지를 코드와 실행 결과로 옮기면 최소한 아래 산출물이 있어야 합니다.

- 회원가입 화면
- 로그인 화면
- auth provider 또는 session state 관리 코드
- `signup`, `login`, `me` API 연결
- 로그인 성공 후 라우트 전이
- 인증 실패/입력 오류 UI

이 산출물이 있어야 다음 페이지에서 admin 기본 조회와 접근 제어를 같은 인증 흐름 위에 올릴 수 있습니다.

# 16. 자주 하는 실수
- 로그인 성공만 구현하고 새로고침 후 복원을 빼먹는다
- 입력 오류와 인증 실패를 같은 메시지로 처리한다
- 세션 상태를 페이지 로컬 state로만 관리한다
- 로그인 후 항상 홈으로만 보내 사용자 맥락을 잃는다
- `me` API 없이 저장된 토큰만 믿고 화면을 연다
- 제출 중 중복 요청을 막지 않는다

이 여섯 가지 실수는 이후 장의 보호 라우트와 주문 흐름을 바로 불안정하게 만듭니다.

# 정리
이 페이지에서 우리는 회원가입, 로그인, 세션 복원을 포함한 실제 사용자 식별 흐름을 `storefront`에 연결하는 방법을 정리했습니다. 또한 인증 화면은 단순 폼이 아니라, `signup`, `login`, `me` API와 앱 전역 세션 상태, 로그인 후 전이, 실패 상태 UX까지 함께 설계해야 한다는 점을 확인했습니다.
