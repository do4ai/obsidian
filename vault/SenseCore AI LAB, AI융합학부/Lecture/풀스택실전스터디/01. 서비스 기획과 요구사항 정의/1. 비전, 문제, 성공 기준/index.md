# 1. 비전, 문제, 성공 기준
이 페이지는 이커머스 서비스를 만들기 전에 팀이 반드시 먼저 고정해야 하는 출발점입니다. 개발 강의에서 가장 흔한 실패는 코드를 너무 빨리 시작하는 것입니다. 무엇을 만들지, 누구의 문제를 해결할지, 어느 수준까지 성공으로 볼지를 정하지 않으면 뒤 장의 DDD 모델링, 화면 설계, API 설계, 인프라 결정이 모두 흔들립니다.

이 강의는 단순히 기능 몇 개를 흉내 내는 것이 아니라, 실제 서비스처럼 기획부터 배포까지 한 줄로 연결해 보는 교재입니다. 그래서 첫 페이지에서 해야 할 일은 기술 선택이 아니라 방향 고정입니다.

# 왜 이 페이지가 먼저 나와야 하는가
개발 강의는 순서가 중요합니다. 특히 이커머스처럼 도메인이 넓은 서비스는 순서를 잘못 잡으면 뒤에서 계속 되감게 됩니다.

예를 들어 아래와 같은 순서로 시작하면 거의 반드시 문제가 생깁니다.

- 프론트 화면부터 먼저 만든다
- 백엔드 API를 감으로 만든다
- 장바구니, 주문, 결제를 붙인 뒤에야 서비스 목표를 생각한다

이 방식은 처음에는 빨라 보이지만, 곧 아래 문제를 만납니다.

- 상품, 재고, 주문, 결제 중 무엇이 핵심인지 경계가 흐려진다
- 고객용 기능과 운영자 기능의 우선순위를 설명할 수 없다
- 화면은 많은데 실제 유저가 끝까지 완료하는 시나리오는 없다
- 성공 기준이 없어서 무엇을 검증해야 하는지 모른다

따라서 이 강의는 반드시 아래 순서를 따릅니다.

1. 서비스 비전과 해결 문제를 고정한다.
2. 누구를 위해 어떤 시나리오를 완성할지 정한다.
3. 그 시나리오를 바탕으로 도메인 경계와 모델을 설계한다.
4. 그 모델을 화면과 API로 번역한다.
5. 그 뒤에야 프론트와 백엔드를 구현한다.

이 페이지는 그 첫 번째 단계입니다.

# 이 페이지에서 최종적으로 정할 것
이 페이지를 읽고 나면 아래 네 가지를 문서로 남겨야 합니다.

- 서비스 비전 한 줄
- 해결하려는 문제 정의
- MVP 범위와 비범위
- 성공 기준과 측정 지표

이 네 가지가 정리되지 않았다면, 아직 다음 페이지로 넘어가면 안 됩니다.

# 강의에서 만들 서비스
이 강의의 대상은 학습용 데모가 아니라, 실제 구조를 가진 작은 이커머스 서비스입니다. 완성 목표는 아래와 같습니다.

- 고객은 회원가입과 로그인을 하고 상품을 탐색할 수 있다
- 고객은 상품을 장바구니에 담고 주문을 생성할 수 있다
- 고객은 결제 성공과 실패 흐름을 경험할 수 있다
- 운영자는 상품, 주문, 고객 상태를 확인할 수 있다
- AI 상담 챗봇은 상품 안내, 주문 안내, FAQ 응답, 상담 전환의 기본 흐름을 제공한다

즉 이 강의는 단순 쇼핑몰 UI 클론이 아니라, 도메인 모델과 운영 흐름이 살아 있는 서비스 구축을 목표로 합니다.

# 1. 비전
비전은 서비스가 왜 존재하는지를 가장 짧게 압축한 문장입니다. 좋은 비전은 멋있는 문구가 아니라, 이후의 구현 우선순위를 결정할 수 있는 문장이어야 합니다.

이 강의에서 사용할 기준 비전은 다음과 같습니다.

> 고객이 빠르게 상품을 탐색하고 신뢰 있게 주문을 완료하며, 운영자는 주문과 상담 흐름을 한 화면에서 관리할 수 있는 실전형 이커머스 서비스를 만든다.

이 문장에는 세 가지 핵심이 들어 있습니다.

- 고객 가치: 빠르게 탐색하고 주문을 완료한다
- 운영 가치: 주문과 상담 흐름을 관리한다
- 구현 방향: 실전형 서비스로 만든다

비전 문장을 쓸 때는 아래 규칙을 지키는 것이 좋습니다.

- 누가 가치를 얻는지 드러나야 한다
- 어떤 핵심 행동을 개선하는지 보여야 한다
- 구현 범위를 지나치게 넓히지 않아야 한다

반대로 아래 같은 문장은 비전으로 부적절합니다.

- 최고의 쇼핑 플랫폼을 만든다
- 최신 기술을 사용한 풀스택 서비스를 만든다
- 모든 사용자가 만족하는 AI 커머스를 만든다

이 문장들은 멋있어 보일 수는 있지만, 무엇을 먼저 구현해야 하는지 결정하는 데 도움을 주지 못합니다.

# 2. 문제 정의
비전이 존재 이유를 말한다면, 문제 정의는 지금 무엇이 불편하고 비효율적인지를 명확하게 적는 단계입니다. 개발자는 문제를 정확히 써야 올바른 모델을 만들 수 있습니다.

이 강의에서 다루는 문제는 크게 세 가지입니다.

## 고객 관점의 문제
- 상품 정보가 흩어져 있으면 원하는 상품을 빨리 찾기 어렵다
- 장바구니와 주문 흐름이 길거나 불안정하면 결제 직전 이탈이 발생한다
- 주문 이후 상태를 확인하기 어렵다면 고객 신뢰가 떨어진다

## 운영자 관점의 문제
- 상품, 주문, 고객 문의가 분리되어 있으면 운영 효율이 떨어진다
- 주문 상태와 결제 상태를 함께 보지 못하면 장애 대응이 늦어진다
- 반복 문의를 사람이 모두 처리하면 운영 비용이 커진다

## 시스템 관점의 문제
- 도메인 경계 없이 기능을 쌓으면 주문과 결제 로직이 쉽게 엉킨다
- 프론트와 백엔드가 같은 시나리오를 기준으로 개발되지 않으면 API와 화면이 자주 어긋난다
- 배포와 검증 흐름이 없으면 서비스가 커질수록 수정 비용이 급격히 커진다

중요한 점은 문제 정의가 단순 불만 목록이 아니라는 것입니다. 문제 정의는 뒤에서 DDD 경계를 자르는 기준이 됩니다.

예를 들어:

- 고객이 상품을 찾기 어렵다 -> Catalog 컨텍스트가 필요하다
- 주문과 결제 상태를 함께 관리하기 어렵다 -> Order와 Payment의 관계를 명확히 정의해야 한다
- 문의 응답 비용이 높다 -> Support 또는 AI Assistant 영역이 필요하다

즉 문제 정의는 이후의 도메인 설계를 위한 원재료입니다.

# 3. 이 강의에서 해결할 범위
비전과 문제를 정했으면, 이제 어디까지 만들지를 결정해야 합니다. 개발 강의는 특히 범위를 자르는 능력이 중요합니다. 범위를 자르지 못하면 완성 대신 확장만 하다가 끝납니다.

## MVP 범위
이 강의에서 반드시 완성할 범위는 아래와 같습니다.

- 고객용 storefront
- 운영자용 admin
- 상품 조회와 상품 상세
- 회원가입, 로그인, 인증 상태 유지
- 장바구니 담기, 수량 변경, 합계 계산
- 주문 생성
- 모의 결제 승인과 실패 처리
- 주문 목록과 주문 상태 확인
- 운영자 주문/상품 대시보드
- AI 상담 챗봇 기본 플로우
- 로컬 개발 환경, 테스트, 빌드, 배포 기본 경로

## 이번 강의에서 제외하는 범위
아래 항목은 실제 서비스에는 중요하지만, 이 교재의 첫 완성 범위에서는 제외합니다.

- 복수 판매자 마켓플레이스
- 실시간 재고 동기화
- 실제 PG 연동
- 쿠폰, 적립금, 멤버십
- 리뷰 추천 알고리즘
- 다국어와 다중 통화
- 대규모 트래픽 최적화

이 제외 범위는 포기가 아니라 순서 조정입니다. 먼저 끝까지 완성 가능한 서비스 하나를 만들고, 마지막 장에서 확장 방향으로 다룹니다.

# 4. 성공 기준
성공 기준은 구현이 끝났는지 판단하는 기준입니다. 개발 강의에서는 특히 이 부분이 중요합니다. 왜냐하면 "기능이 얼추 되는 것 같다"는 말은 검증 기준이 아니기 때문입니다.

이 강의의 성공 기준은 세 층위로 나눕니다.

## 4.1 사용자 성공 기준
- 신규 사용자가 상품 조회에서 주문 생성까지 한 흐름을 완료할 수 있다
- 로그인한 사용자가 장바구니와 주문 상태를 안정적으로 확인할 수 있다
- 사용자가 AI 상담 챗봇을 통해 상품 문의 또는 주문 문의를 시작할 수 있다

## 4.2 운영 성공 기준
- 운영자가 상품과 주문 현황을 한 화면에서 확인할 수 있다
- 결제 실패 또는 주문 이상 상태를 식별할 수 있다
- 반복 문의의 일부를 챗봇이 흡수하고, 필요 시 사람 상담으로 넘길 수 있다

## 4.3 개발 성공 기준
- DDD 기반 경계가 문서와 코드에 모두 반영된다
- 프론트와 백엔드가 동일한 유스케이스를 기준으로 연결된다
- 로컬 개발 환경이 재현 가능하다
- 주요 시나리오에 대한 테스트와 검증 기록이 남아 있다
- 배포 후 기본 사용자 흐름을 다시 확인할 수 있다

이 기준은 단순 설명으로 끝내지 말고, 실제로 측정 가능한 항목으로 바꿔야 합니다.

# 5. 성공 기준을 측정 가능한 문장으로 바꾸기
좋은 기획 문서는 추상적인 문장을 남기지 않습니다. 반드시 관찰 가능한 결과로 번역합니다.

예를 들어 아래처럼 바꾸어 써야 합니다.

| 추상적 표현 | 측정 가능한 표현 |
| --- | --- |
| 사용자가 쉽게 상품을 찾는다 | 사용자가 카탈로그에서 상품 상세로 이동할 수 있다 |
| 주문이 잘 된다 | 로그인 사용자가 장바구니에서 주문 생성까지 완료할 수 있다 |
| 운영이 편하다 | 운영자가 주문 목록에서 상태별 주문 수를 확인할 수 있다 |
| 챗봇이 도움을 준다 | 챗봇이 FAQ 응답 후 상담 전환 버튼을 제공할 수 있다 |
| 서비스가 안정적이다 | 핵심 시나리오가 로컬과 배포 환경에서 재현된다 |

강의에서는 이 문장을 나중에 그대로 테스트 시나리오와 검증 체크리스트로 바꿉니다. 즉 지금 적는 성공 기준은 마지막 장의 검증 문서까지 이어집니다.

# 6. 이 페이지의 결정이 뒤 장에 어떻게 연결되는가
이 페이지는 혼자 존재하지 않습니다. 뒤의 모든 장이 여기서 정한 내용을 이어받습니다.

## DDD 모델링으로의 연결
- 고객 탐색 문제 -> Catalog 컨텍스트 설계
- 주문 완료 문제 -> Cart, Order, Payment 컨텍스트 설계
- 운영 효율 문제 -> Admin/Operations 관점 설계
- 반복 문의 문제 -> Support Assistant 경계 설계

## 화면 기획으로의 연결
- 고객 핵심 시나리오 -> 상품 목록, 상세, 장바구니, 주문, 결제 화면
- 운영 시나리오 -> 대시보드, 주문 목록, 상품 관리 화면
- 상담 시나리오 -> 챗봇 패널, FAQ 응답, 상담 전환 UI

## 기술 구현으로의 연결
- 성공 기준 -> API 우선순위
- MVP 범위 -> 라우트와 모듈 범위
- 비범위 -> 이번 장에서 만들지 않을 기능 정리

즉 이 페이지는 “좋은 말 적는 페이지”가 아니라, 뒤의 설계와 구현을 묶는 기준 페이지입니다.

# 7. 실습용 산출물 템플릿
이 장을 직접 따라 하는 학습자는 아래 템플릿을 채워 두는 것이 좋습니다.

## 비전 문장
`누가 어떤 행동을 더 빠르고 안정적으로 끝낼 수 있게 만들 것인가?`

예시:

`고객이 상품 탐색부터 주문 완료까지 빠르게 끝내고, 운영자가 주문과 상담 흐름을 효율적으로 관리할 수 있는 이커머스 서비스를 만든다.`

## 문제 정의
- 고객은 무엇이 불편한가?
- 운영자는 무엇이 비효율적인가?
- 현재 시스템은 어디에서 쉽게 깨지는가?

## MVP 범위
- 이번 강의에서 반드시 만드는 기능
- 다음 버전으로 넘기는 기능

## 성공 기준
- 사용자는 무엇을 끝까지 완료할 수 있어야 하는가?
- 운영자는 무엇을 볼 수 있어야 하는가?
- 개발자는 무엇을 재현 가능하게 만들어야 하는가?

# 8. 이 페이지를 읽고 반드시 남겨야 하는 결정
다음 체크리스트를 모두 채웠다면 이 페이지의 목적은 달성된 것입니다.

- 서비스 비전 문장을 한 줄로 적었다
- 고객 문제와 운영 문제를 분리해서 적었다
- 이번 강의의 MVP 범위를 정했다
- 이번 강의에서 하지 않을 범위를 적었다
- 성공 기준을 측정 가능한 문장으로 바꿨다
- 이 결정이 다음 장의 DDD 모델링으로 이어진다는 것을 이해했다

# 9. 자주 하는 실수
- 비전을 기술 스택 소개처럼 적는다
- 문제 정의 대신 원하는 기능 목록만 적는다
- 성공 기준 없이 “대충 되면 됨” 상태로 넘어간다
- 범위를 자르지 못해 결제, 쿠폰, 추천, 리뷰, 멀티벤더를 한 번에 넣으려 한다
- 고객 관점과 운영 관점을 섞어서 도메인 경계를 흐리게 만든다

이 다섯 가지 실수는 뒤 장 전체를 어렵게 만듭니다. 특히 DDD는 문제와 경계가 선명해야 힘을 발휘하므로, 이 페이지를 대충 쓰면 뒤에서 모델이 무너집니다.

# 정리
이 페이지에서 한 일은 “쇼핑몰 하나 만들어보자”를 실제 개발 가능한 목표로 바꾸는 작업입니다. 우리는 누구를 위해 무엇을 만들고, 어디까지 만들고, 무엇을 성공으로 볼지를 먼저 고정했습니다. 이제 다음 장에서는 이 기준을 바탕으로 바운디드 컨텍스트, 애그리거트, 엔티티, 값 객체, 도메인 서비스 같은 DDD 구조로 내려갑니다.
